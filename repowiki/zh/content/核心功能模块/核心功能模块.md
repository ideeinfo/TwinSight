## 设施管理

设施（Facility）是TwinSight平台中的顶层组织单元，代表一个建筑或园区项目。设施管理模块提供了对设施的创建、查询、更新、删除以及关联模型文件管理的能力，实现了"设施→模型→空间/资产"的层级数据隔离。

### 数据模型

设施在数据库中对应`facilities`表，与`model_files`表建立一对多关系。每个设施包含编码、名称、描述、地址、封面图片等基本信息。

```mermaid

integer id PK
varchar facility_code UK
varchar name

varchar status

timestamp updated_at
}
MODEL_FILES {
integer id PK
integer facility_id FK
varchar file_code UK
varchar title
varchar status
boolean is_active
timestamp created_at
timestamp updated_at
}
FACILITIES ||--o{ MODEL_FILES : "1:N"
```

**设施统计聚合**

`facility.js`模型提供了丰富的聚合查询功能，`getAllFacilities`和`getFacilityById`方法不仅返回设施基本信息，还通过子查询统计关联数据：
- `model_count`：关联的模型文件数量
- `view_count`：所有模型下的视图总数
- `asset_count`：所有模型下的资产总数
- `space_count`：所有模型下的空间总数
- `iot_count`：启用的IoT配置数量
- `document_count`：设施关联的文档数量
- `default_model_id`：默认模型ID（按display_order排序）

### CRUD操作

设施管理通过`/api/v1/facilities`端点提供完整的CRUD操作，受JWT身份验证和基于权限的授权保护。

### 前端实现

**FacilitiesView.vue**

设施列表页面提供卡片式布局展示所有设施，每个卡片显示：
- 设施封面图片（如有）
- 设施名称和描述
- 统计信息（模型数、资产数、空间数）
- 操作按钮（查看、编辑、删除）

**HomeDashboardView.vue**

用户仪表盘首页展示设施概览，包括：
- 用户拥有的设施总数
- 各设施的关键指标
- 快速导航到设施详情

设施状态在组件内通过局部响应式状态管理，不经过Pinia store，直接调用`api/facilities.ts`服务获取数据。

### 权限控制

设施管理使用以下权限进行细粒度控制：
- `FACILITY_READ`：查看设施列表和详情
- `FACILITY_CREATE`：创建新设施
- `FACILITY_UPDATE`：更新设施信息
- `FACILITY_DELETE`：删除设施

## 3D模型查看器（Viewer）

TwinSight的3D模型查看器是系统的核心交互界面，基于Autodesk Forge Viewer技术构建，用于可视化和操作建筑信息模型（BIM）。查看器集成在`AppViewer.vue`组件中，通过`MainView`子组件加载和渲染SVF格式的3D模型。

### 模型加载（SVF）

系统通过静态文件服务提供对SVF模型文件的访问。后端在`server/index.js`中配置了`/models`路径的静态文件服务，并设置了必要的CORS和MIME类型头，确保Forge Viewer的Web Worker能够正确加载二进制模型文件。

### 交互控制与信息叠加

查看器提供了丰富的交互功能，包括旋转、缩放、平移、构件隔离和属性查看。用户可以通过左侧的`AssetPanel`或`LeftPanel`选择资产或空间，右侧的`RightPanel`会显示其详细属性。`MainView`组件负责处理模型加载、事件监听和视图状态管理。

信息叠加通过将数据库中的资产和空间数据与模型中的`dbId`进行映射来实现。当模型加载完成后，系统会将从数据库获取的资产和空间列表与模型中的构件进行匹配，从而实现数据与模型的联动。

## 资产（Asset）与空间（Space）管理

资产与空间是系统中的核心数据实体，分别代表建筑内的设备和区域。它们属于特定设施下的模型文件，形成"设施→模型→空间/资产"的层级结构。其数据模型和CRUD操作通过RESTful API进行管理。

## 文档管理

文档管理功能允许用户上传、分类和管理与资产或空间关联的文档（如PDF、Word、Excel等），并能将文档元数据同步至Open WebUI知识库。

### `document-sync-service`同步至Open WebUI

`document-sync-service.js`是一个后台服务，负责将系统中的文档自动同步到Open WebUI的知识库中。该服务每5分钟运行一次，查找未同步的文档，并根据文档关联的资产或空间，找到对应的知识库进行上传。

**同步逻辑流程**
```mermaid
flowchart TD
A["启动同步服务"] --> B["检查Open WebUI健康状态"]
B --> C{"服务可用?"}
C --> |否| D["跳过本次同步"]
C --> |是| E["查询未同步的文档"]
E --> F{"存在未同步文档?"}
F --> |否| G["结束"]
F --> |是| H["遍历每个文档"]
H --> I["查找关联的知识库ID"]
I --> J{"找到知识库?"}
J --> |否| K["跳过"]
J --> |是| L["上传文档到Open WebUI"]
L --> M{"上传成功?"}
M --> |是| N["记录同步成功状态"]
M --> |否| O["记录同步失败状态"]
N --> P["统计成功数量"]
O --> Q["统计失败数量"]
H --> R["等待500ms"]
R --> H
H --> S["输出同步统计"]
S --> T["结束"]
```

**同步逻辑**
1.  **查找未同步文档**：查询`documents`表中未在`kb_documents`表中记录的文档。
2.  **确定知识库**：根据文档关联的`asset_code`或`space_code`，查找其对应的`file_id`，再通过`file_id`找到`knowledge_bases`表中的`openwebui_kb_id`。
3.  **上传文档**：调用`openwebui-service.js`的`uploadDocument`方法，将文档上传到Open WebUI。
4.  **记录状态**：无论成功或失败，都在`kb_documents`表中记录同步状态，防止重复尝试。

**API调用示例**

**手动触发同步（内部调用）**
```javascript

await triggerSync();
```

## 时序数据（Timeseries）模块

时序数据模块负责连接InfluxDB数据库，执行实时查询和历史数据展示，为用户提供设备运行状态的可视化分析。

### 连接InfluxDB

系统通过`@influxdata/influxdb-client`库与InfluxDB进行交互。连接配置（URL、Token、Org、Bucket）在`server/config/index.js`中定义，并在`timeseries.js`路由中初始化`InfluxDB`客户端。

### 实时查询与历史数据展示

系统提供了多种API来查询时序数据，包括获取最新值、查询时间范围内的数据以及获取统计数据。这些数据被用于在3D模型上叠加温度标签和生成图表。

```mermaid

B --> C["后端构建Flux查询"]
C --> D["执行InfluxDB查询"]
D --> E["处理查询结果"]
E --> F["返回JSON数据"]
F --> G["前端更新UI (标签/图表)"]
```

**用户操作流程图**
